Scientific Foundation

The Science of Symbiotic AI

ODEI's L5 (Symbiotic AI) is not a marketing term—it is a legitimate extension of established agent taxonomies, grounded in four decades of Human-AI Teaming research.

01 // TAXONOMIC POSITION

Extending AIMA Agent Classification

Russell & Norvig's canonical taxonomy (AIMA, 1995–2020) defines five agent classes: Reactive, Model-based, Goal-based, Utility-based, and Learning agents. Symbiotic AI extends this taxonomy by adding invariants that no existing class captures.

AIMA Classes (L1–L4)
  • Single-agent optimization
  • Human as operator or supervisor
  • Static goal specification
  • Unidirectional adaptation
→
Symbiotic AI (L5)
  • Joint human-AI system optimization
  • Human as co-participant
  • Dynamic goal co-evolution
  • Bidirectional co-adaptation

L4 agents may have memory and autonomy, but lack mandatory User Model, bidirectional adaptation, and divided subjectivity. L5 is a new class—different invariants, different engineering requirements.

Six Research Traditions

A1

Human-AI Teaming

Treating AI as teammate, not tool. Shared mental models and mutual comprehension.

03 // DEFINING INVARIANTS

What Makes L5 a New Class

Symbiotic AI is defined by four invariants absent from L1–L4 agent classes:

01

Mandatory User Model

The agent maintains and updates a persistent model of the human's goals, preferences, cognitive state, and behavioral patterns.

02

Bidirectional Policy Update

Agent policy updates based on outcomes + human feedback. Not just reward signals, but explicit preference learning.

03

Co-Adaptation Loop

Human ↔ Agent mutual adjustment. Agent learns preferences; human learns to formulate, calibrate trust, and evolve processes. Both modify behavior. The system evolves together.

04

Divided Subjectivity

Agent has no goals outside the human's framework, but possesses execution autonomy. Source of goals ≠ source of action.

Memory

Assistants have chat history. ODEI has memory architecture. The difference: chat history is theirs, resets on their terms, and doesn't affect how the system behaves. Memory in ODEI is yours, persistent, and directly shapes every decision.

THREE ARCHITECTURAL PROPERTIES

These are properties 9–11 from the Roadmap page. They define what separates L5 Symbiotic AI from L4 Agentic systems.

09

Persistent Memory

Cross-session memory that affects decisions. Not just "remember what we talked about"—but facts, beliefs, patterns, and outcomes that shape how ODEI reasons about your world. Every interaction updates the model. Nothing is lost between sessions.

10

User Model

Persistent representation of you—goals, preferences, constraints, communication style, decision patterns. Not a profile you fill out. A model that emerges from observation and explicit feedback. ODEI doesn't guess who you are. It learns.

11

Policy Update

Outcomes change strategy, not just state. When you approve, deny, or override—ODEI doesn't just record the action. It updates the operating rules. Your corrections become versioned policy. The system gets better at being useful to you specifically.

THREE OWNERSHIP PRINCIPLES

Memory without ownership is surveillance. These principles ensure your memory remains your asset.

P1

Portable

Your memory is not locked to a model or vendor. Export anytime. Switch providers. Keep your accumulated context. The value you build stays with you, not with the platform.

P2

Versioned

Every change is tracked. Policies have versions. Facts have timestamps. If something goes wrong, roll back. If you want to understand why ODEI behaves a certain way, trace the history.

P3

Provenance

Every fact has a source. Every belief has evidence. Every policy has an origin. No "I just know"—full traceability from memory to decision to action. Audit everything.

THE BOTTOM LINE

Memory is what makes L5 possible. Without persistent memory, no user model. Without user model, no adaptation. Without adaptation, no symbiosis. Just another stateless assistant pretending to know you.

Structural Barriers to L5

⟳
Stateless Architecture

Optimized for API calls. Each request = new context. L5 needs persistent stateful agents that live between sessions.

☁
Centralized Data = Their Asset

Your data on their servers = their value. L5 requires user-owned, local-first data. This kills their business model.

⚖
Policy Update = Liability

If AI learns from your feedback, it might learn "wrong". Public companies avoid this legally.

⊘
Deep User Model = Privacy Risk

True user model (goals, patterns, decisions) is a regulatory nightmare for public companies.

$
Revenue Model Conflict

L4 = many API calls per task (good for billing). L5 = efficiency, fewer calls, more value. Economically backwards.

⚡
Scale vs Depth

Optimized for millions with one model. L5 needs deep per-user personalization. Different architecture entirely.

L5 isn't a feature they can add. It requires local-first infrastructure, user-owned data, persistent co-adaptation, and aligned incentives. Their architecture and business model are structurally incompatible.

Effective date: December 18, 2025

This Privacy Policy explains how ODEI ("we", "us", "our") collects, uses, and protects information when you use odei.ai and ODEI services. ODEI is designed as a local-first, privacy-by-default system with full transparency and user control.

Core Principles

  • Local-first architecture. Your data lives on your devices by default. Cloud sync is opt-in and encrypted.
  • No data monetization. We do not sell, rent, or trade your personal data. Ever.
  • Full provenance. Every piece of data has a source, timestamp, and you can trace how it's used.
  • Revocable permissions. Disconnect integrations and delete data at any time with immediate effect.
  • Audit trail. All system actions are logged with receipts—you can verify what ODEI did and why.

Data We Process

  • Account data (optional). Email and authentication identifiers only if you create an account.
  • Your content. Notes, tasks, goals, calendar events, and other inputs you create. Stored locally unless you enable sync.
  • Connected services. With explicit permission: calendar, email, health data. You control what's connected and can disconnect instantly.
  • System metadata. Minimal diagnostics for reliability. Analytics are anonymized and opt-in only.
  • Beliefs and facts. ODEI maintains a working model of your context—all derived from data you provide, with clear provenance.

How We Use Data

  • Operational subjectivity. To maintain ODEI's working state and generate actionable proposals.
  • Verification. To confirm actions were executed correctly and provide receipts.
  • Learning loop. To improve recommendations based only on your explicit feedback (approve/deny/override).
  • Security. To detect and prevent unauthorized access.

AI Model Interactions

  • Minimal data transmission. When using cloud AI models, we send only what's necessary for the specific task.
  • No training on your data. We configure providers for no-retention, no-training where available.
  • Model-agnostic. You can choose which AI providers to use or run models locally.

Your Rights

  • Access. Request a complete export of your data at any time.
  • Deletion. Delete any or all data from device and cloud sync.
  • Portability. Export in standard formats (JSON, CSV).
  • Correction. Fix any inaccuracies in your data.
  • Objection. Disable specific processing or features.

Security

  • Encryption. TLS in transit. AES-256 at rest for cloud data. End-to-end encryption for sensitive vaults.
  • Access control. Principle of least privilege. Regular security audits.
  • Incident response. Report vulnerabilities to security@odei.ai.

Data Retention

  • User-controlled. You decide what to keep and what to delete.
  • Fact freshness. ODEI automatically tracks data age and marks stale information.
  • System logs. Minimal retention (30 days) for debugging and security.

Children

  • ODEI is not intended for users under 16. We do not knowingly collect data from children.

Changes

  • Material changes will be communicated with advance notice. Continued use constitutes acceptance.
